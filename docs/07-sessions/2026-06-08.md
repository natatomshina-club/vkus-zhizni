## Баг #1 — subscription_started_at сбрасывался рекуррентом

- **Корень:** PAY-блок webhook безусловно перезаписывал поле при каждом платеже. CloudPayments шлёт всё как `OperationType='Payment'`, RECURRENT-блок не используется на проде
- **Жертвы:** 16 строк в БД по аудиту (`subscription_started_at::date = last_payment_at::date AND started > created + 5 days`); 13 откатываемых; 9 восстановлено вручную через SQL
- **Фикс:** `payUpdate.subscription_started_at = existing.subscription_started_at ?? new Date().toISOString()` — поле пишется один раз
- **Открытие:** «Бриллиант» — вычисляемый уровень через `getEffectiveMonths(started_at, plan)` в `src/lib/webinars.ts`, в БД отдельного поля нет; исторические `UPDATE tariff='Бриллиант'` эффекта не давали
- **Открытие про is_manual_subscription:** флаг защищает `subscription_ends_at` от webhook. Для VIP, платящих через CP-рекуррент, поднимать опасно — даты не будут продлеваться. Поднимать только для ручного продления

## Баг #2 — видео не отображалось в курсе

- **Корень:** `lesson_type` был управляющим флагом в `renderLessonContent` (`CoursePageClient.tsx`): видео рендерилось только при `type === 'video'`, текст — только при `type === 'text'`. PATCH-роут `/api/admin/intro-lessons/[id]` не обновлял `lesson_type` при сохранении `video_url` → урок «Видео об Умной кухне» создан как `text`, ссылка добавлена позже, тип не сменился
- **Фикс:** рендер переведён на data-driven: видео показывается при наличии `videoId`, текст при наличии `textContent`, независимо от `lesson_type`. PATCH-роут дополнительно выставляет `lesson_type='video'` при сохранении непустого `video_url` (консистентность данных + корректная иконка в админке)
- **Бонус:** теперь возможны гибридные уроки (видео + текст-конспект под ним)
- **SQL на проде:** `UPDATE intro_lessons SET lesson_type='video' WHERE id='53111e02-36ca-4256-b792-1db958f7e4e6'` (1 затронутый урок, `text_content = NULL`)

## Уроки сессии

- При обработке рекуррентных платежей CloudPayments — все транзакции идут как `OperationType='Payment'`, RECURRENT-блок мёртвый. Это нужно помнить при работе с webhook'ами CP.
- Перед массовым откатом данных — точный аудит с проверкой гипотезы по нескольким полям (анализ gap между `started_at` и `last_payment_at`), а не только по одному критерию.
- `lesson_type` как управляющий флаг для рендера — антипаттерн. Рендер должен быть data-driven по наличию полей, а тип — только метка UI (иконка в списке).
- Когда участница говорит «слетел Бриллиант» — это не «потерян manual статус», а сброс `subscription_started_at`. Два разных понимания, важно различать на старте диагностики.
