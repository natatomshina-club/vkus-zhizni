## 2026-06

### Обезврежен legacy Vercel deployment

**Когда:** 2026-06-06

С апреля параллельно основному коду на Beget работал старый Vercel-проект `vkus-zhizni` (deployment от 2 апреля, коммит `5560cb4`): 5 Cron Jobs запускались ежедневно на старом коде — `birthdays/check`, `cleanup-media`, `cleanup-posts`, `check-subscriptions`, `subscription-reminders`. Старый `subscription-reminders` шёл через Resend API напрямую; шаблон письма «Ваша подписка истекла, и доступ в клуб временно приостановлен» жил только в Vercel-deployment'е — в репо его не было, поэтому grep ничего не находил. DNS Cloudflare → Beget (IP 155.212.130.228) — Vercel трафик не обслуживал, но crons работали независимо на собственных vercel.app-URL.

В прошлой сессии (R54, 22 мая) был disconnect Git от Vercel — но это останавливает только auto-deploy на push, **не сам deployment и не crons**. Обнаружен через Resend dashboard: поле `From: noreply@nata-tomshin.ru` в конкретном письме указало на старый Vercel за минуту.

**Решение:** Vercel Cron Jobs disabled через UI; `vercel.json` удалён из репо; пакет `resend` убран; `track-email/route.ts` удалён. GoTrue OTP подтверждён — идёт через `smtp.resend.com:465` как SMTP relay (`SMTP_PASS` в `/opt/supabase/docker/.env`), это отдельный канал, не затронут.

**Урок:** disconnect ≠ отключение. При выходе с любой PaaS-платформы явно останавливать crons/scheduled tasks отдельно (Pause Project или Delete Project).

**Источники:** [[07-sessions/2026-06-06]], [[07-sessions/SESSION_2026-06-06_VAULT_HANDOFF]]

### R02. Правило: никаких prod-изменений без явного «делай»

**Когда:** 2026-05-17
**Что решено:** Анализ в чате с Claude.ai — контекст, не команда. Любые изменения на проде (сервер, `.env`, БД, docker) только после явного «делай» / «go» от Наташи.

### R03. OTP клуба идёт через Resend (GoTrue), не через mailer.ts

**Когда:** 2026-05-17
**Что решено:** Клубный OTP (`/auth`, `/join`) идёт через `supabase.auth.signInWithOtp()` → Supabase GoTrue → smtp.resend.com — **не** через `mailer.ts`/Beget. Бесплатный план Resend: 100 emails/day, 3 000/month. При росте клуба упрёмся в лимит — логин сломается.
**Почему:** Обнаружено при расследовании неизвестных писем «Доступ приостановлен». До этого считали, что весь email идёт через `mailer.ts`.
**Альтернативы:** Немедленная миграция GoTrue → Beget SMTP — отклонено (риск потери входа для участниц). Отложено как открытая задача.
**Источники:** [[07-sessions/2026-05-17]]
**См. также:** R47 (mailer.ts создан именно для изоляции провайдера), [[08-roadmap/todo]] (миграция OTP — открытая задача)

### R05. PDF race condition: useSearchParams в Suspense

**Когда:** 2026-05-17
**Что решено:** `useSearchParams()` внутри `Suspense` может вернуть `null` до монтирования при client-side навигации. Правило: `searchParams` для PDF-страниц читать как пропсы в Server Component, не через хук на клиенте.
**Почему:** `PdfViewerPage` использовал `useSearchParams()` внутри `Suspense` — url возвращался пустым, iframe не рендерился после client-side навигации.
**Альтернативы:** не задокументировано

### R06. nutrition: нормализация ё→е в коде и БД

### R21. Система уровней: getEffectiveMonths, даты, Infinity→999, бонус halfyear

**Когда:** 2026-04-11 (рефакторинг) / 2026-04-14 (правила зафиксированы)
**Что решено:** Единая функция `getEffectiveMonths(subscription_started_at ?? created_at, subscription_plan)`. Правила:
- Дата отсчёта — `subscription_started_at`, не `created_at`
- Квота Бриллиант: `Infinity` заменён на `999` (Infinity ломал JSON-сериализацию)
- Участницы «Полгода» получают +6 бонусных месяцев к стажу: день 1 → «Уже своя», 3 реальных месяца → «Легенда», 6 → «Бриллиант»
- UI показывает «Безлимитный доступ» вместо числа 999

**Почему:** Логика уровней была в 3 местах с расхождениями. `Infinity` ломал JSON и давал проблемы со сравнениями в UI. «Полгода»-участницы теряли бонус стажа.
**Альтернативы:** не задокументировано
**Источники:** PROJECT_BRAIN_v14, PROJECT_BRAIN_v15 (`~/Desktop/Клуб разработка/`)
**См. также:** R04 (subscription_plan не писался для halfyear), R08 (is_manual_subscription)

### R29. RegExp /g + lastIndex: непредсказуемый .test()

**Когда:** 2026-04-11

**Альтернативы:** Создавать новый `RegExp(...)` при каждом вызове — работает, но паттерн `split` надёжнее и не требует хранить строку паттерна отдельно.

### R36. UNIQUE constraints в upsert: pending_refs и webinar_selections

**Когда:** 2026-04-07 (pending_refs) / 2026-04-14 (webinar_selections)
**Что решено:** При upsert в Supabase необходим UNIQUE constraint на конфликтующее поле. `pending_refs`: `UNIQUE(email)` + `upsert({onConflict: 'email'})`. `webinar_selections`: `UNIQUE(member_id, webinar_id)` + `upsert` с `ignoreDuplicates: true`. Без UNIQUE — повторная попытка даёт 500.
